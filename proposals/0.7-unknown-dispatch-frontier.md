# Proposal — spec 0.7: the unresolved-dispatch frontier on `callers`

**Status: design proposal (unreleased).** Anchors the ⟨0.7⟩ work; nothing here is in the contract until
it ships across all engines and conformance gates it (the 0.6 discipline: spec → reference impl → roll
to engines → conformance → move header + declarations).

## Motivation

`callers <fn>` (and `whatif`/`impact`, which share its reverse reachability) returns the **confirmed**
transitive callers — the blast radius if `fn` gained an effect. On dispatch-heavy OO code that set is a
**silent lower bound**: when a call dispatches over a project abstraction with more than
`CHA_FANOUT_LIMIT` implementors, an engine bounds the CHA — it drops the candidate edges and discloses
the call site as `Unknown` with `unknownWhy = dispatch-broad[-ext]:OWNER.M` (sound: it refuses to
fabricate or over-propagate across a wide dispatch). A function that reaches `fn` **only** through such
a dispatch is therefore *correctly disclosed as `Unknown` in its own report row*, but is **absent from
`callers`** — so the blast-radius answer doesn't reveal that it stops at an unresolved dispatch.

Observed on real code (the candor-java token/speed eval, batch 3): a scheduled-task base class whose
abstract `performTask()` has 15 implementors — its `execute()` is `{…,Unknown}` /
`dispatch-broad:…performTask`, and one implementor reaches the target, so `execute` *may* reach it, but
`callers` (rightly) doesn't assert it. This is **not** an under-report (the function is `Unknown`-flagged,
not silent-pure) — it is `callers` not disclosing its own dispatch-bounded incompleteness.

The reference engine ships a prototype (`candor-java 0.5.43`, `callers --include-unknown`) that discloses
this frontier, but matches candidates **by dispatch-method simple name** (the report has no type
hierarchy), so it *over-lists* (e.g. an unrelated `setText` dispatch flagged because some `setText`
override reaches the target). Fine as a labelled "cannot confirm" CLI affordance; **too imprecise to
freeze as a cross-engine contract**. This proposal makes it precise.

## The precise design (flood-free)

The naive precise approach — record, per broad-dispatch site, the dropped candidate impls — re-encodes
the exact flood bounded-CHA exists to prevent (a project iface with hundreds of impls → hundreds of
edges × many sites). **Reject it.** Instead:

### 1. Emit a compact type-hierarchy map ⟨0.7⟩

Each engine emits, alongside the report, a **hierarchy** of project types it loaded:

```
type  →  [ direct supertypes + implemented interfaces ]   // project types only; O(classes), not O(edges)
```

This is small (one entry per class, a handful of supertypes each), reusable, and — crucially — lets a
*query* resolve "is X an override of OWNER.M?" precisely without the engine storing exploded candidate
sets. Placement options (decide before reference impl):
- **(a) extend the §2.2 sidecar** — add a sibling `"hierarchy"` object to the call-graph sidecar JSON
  (the sidecar is already the structural-query companion; keeps one artifact). *Recommended.*
- (b) a new `<prefix>.hierarchy.json` sidecar (fully separate, fully optional).
- (c) a per-report field — rejected (bloats the §2 report; the hierarchy is a query aid, not an effect fact).

### 2. Precise frontier semantics ⟨0.7⟩

`callers fn` with the frontier requested returns, in addition to `transitive`:

> **possibleViaUnknownDispatch**: every function `F` such that `F` carries
> `unknownWhy = dispatch-broad[-ext]:OWNER.M`, `F` is **not** already a confirmed transitive caller, and
> there exists a confirmed reacher `R ∈ transitive ∪ {fn}` that is an **override of `OWNER.M`** —
> i.e. `R`'s method simple-name = `M`'s simple-name **AND** `R`'s declaring type is a subtype of `OWNER`
> per the hierarchy map. Each entry names `F` and the dispatch method `OWNER.M` it travels.

The subtype check (vs the prototype's simple-name-only match) is what removes the false positives: an
unrelated `setText` dispatch no longer matches unless its `OWNER` actually sits above a reaching override.
Transitive callers of an `F` are themselves possible reachers; an engine MAY expand transitively (mark
the whole reverse cone of `F` as possible) or report only the frontier `F`s — **decide**: reporting only
the frontier `F`s (the dispatch sites) is the more informative + smaller set and is the recommended shape.

### 3. Surfacing — a modifier, not always-on ⟨0.7⟩

Keep the core `callers { of, direct, transitive }` contract unchanged; the frontier is requested with a
standard **`--include-unknown`** modifier, adding `possibleViaUnknownDispatch: [ { fn, viaDispatchOn } ]`
to the output. Rationale: the confirmed set is the load-bearing answer; the frontier is a deliberate
"show me the lower-bound's edge" opt-in, and an always-on field would bloat every blast-radius call.

### 4. Flood control ⟨0.7⟩

Only `dispatch-broad`/`dispatch-broad-ext` sites (already the project-meaningful signal — the engine
dropped *project* impls) contribute. The hierarchy keeps resolution compact. If a single `OWNER.M`
resolves to a very large override set that intersects the reachers, the entry is still **one** line
(`F via OWNER.M`), so there is no flood in the output. (No cap needed; the output is per-dispatch-source,
not per-candidate.)

## Conformance ⟨0.7⟩

Add to `conformance/run.sh` PART 5: a fixture with a project abstraction `Base.op()` over
`> CHA_FANOUT_LIMIT` implementors, exactly one of which reaches a target `T`, and a caller `F` that
dispatches `Base.op()`. Assert every engine's `callers T --include-unknown` yields the **same**
`possibleViaUnknownDispatch` set (`[{fn:F, viaDispatchOn:Base.op}]`) and the **same** confirmed
`transitive`. The precise (hierarchy-resolved) definition makes this differential-gradeable; the
prototype's simple-name match would not agree across engines.

## Cross-engine applicability (finding — 2026-06-19, during rollout)

The frontier is a **class/protocol-dispatch** concept, and the engines' `unknownWhy` vocabularies differ
by language model (this is already logged: candor-rust `BACKLOG.md` item 1d — the dispatch vocabulary
diverges four ways; the spec's canonical set is `reflect:/native:/dispatch:/callback:`, so Java's
`dispatch-broad:` is itself an **off-spec** extension):

- **candor-java** — `dispatch-broad:OWNER.M` (bounded-CHA over many implementors). Reference impl done.
- **candor-ts / candor-swift** — emit `dispatch:Type.member` (an unresolved virtual/protocol dispatch)
  and have a class/protocol hierarchy. **The frontier applies here** — these are the real next ports —
  but it keys off `dispatch:`, not `dispatch-broad:`.
- **candor-rust** — indeterminacy is `callback:unresolved call` / `ffi:extern fn` only; **no type-dispatch
  reason and no class hierarchy**. The frontier is **legitimately empty** for Rust (a callback/FFI
  unknown does not resolve to a target-specific override). Rust participates by emitting
  `possibleViaUnknownDispatch: []` consistently — N/A by language model, not a gap to fill.

**Consequence — a prerequisite for contracting:** the frontier keys off the dispatch `unknownWhy`
reason, and those reasons aren't harmonized. Before this can be a conformance-EQUAL contract, the
dispatch vocabulary must be canonicalised (a `dispatch:OWNER.M` form all class/protocol engines emit,
with the bounded-CHA "many impls" nuance carried in an agreed way), and the Java reference reconciled to
it. That is the standalone work of `BACKLOG.md` 1d (an `unknownWhy` vocabulary spec decision), which this
feature now depends on. Recommended order: **(1) harmonise the dispatch `unknownWhy` vocabulary → (2)
port the frontier to ts + swift on the canonical reason + reconcile Java → (3) Rust emits empty → (4)
conformance + stamp 0.7.**

## Rollout sequence (the 0.6 discipline)

1. **Spec** — land the ⟨0.7⟩ parts in `SPEC.md` (§2.2 hierarchy map; §3.1 `callers` `--include-unknown`
   + `possibleViaUnknownDispatch`; §4 note that the frontier is a *disclosed lower-bound expansion*, never
   an assertion). Header stays `0.6` until rollout completes.
2. **Reference impl** — candor-java: emit the hierarchy map; replace the 0.5.43 simple-name match with the
   precise subtype-resolved frontier.
3. **Roll** — candor-scan/rust, candor-ts, candor-swift: emit the hierarchy; implement the precise frontier.
4. **Conformance** — the PART 5 fixture above; EXIT 0 across engines.
5. **Stamp 0.7** — move the `SPEC.md` header + every engine's declared spec version to `0.7` together.

## Decisions to confirm before engine work

- **Hierarchy placement**: (a) extend the §2.2 sidecar [recommended] vs (b) a separate sidecar.
- **Frontier shape**: report only the dispatch-source `F`s [recommended] vs transitively expand their cones.
- **Surfacing**: `--include-unknown` modifier [recommended] vs always-on field.

(The candor-java 0.5.43 prototype stays as-is meanwhile — an opt-in, clearly-labelled approximation — and
is replaced by the precise version when the reference-impl step lands.)
